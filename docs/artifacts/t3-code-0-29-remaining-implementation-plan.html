Change-set rules

These constraints keep the large rebuild reviewable and reversible.

Start from a clean worktree and name the affected domain.
Keep contracts schema-only and identifiers unchanged.
Add focused compatibility tests before moving the next owner.
Keep central scope enforcement and RPC instrumentation.
Test both UI generations until the cutover phase.
Do not mix server extraction with presentation rewrites.
Do not move native modules into shared runtime.
Preserve cached snapshots during reconnect and mark them stale.
Never delete project or thread history in migration code.

Risk register and decision points

Highest technical risks

  1. Duplicate state ownership: stop UI expansion if web/mobile bypass the shared workspace model.
  2. Stream handoff regressions: orchestration extraction stays last and requires replay/concurrency tests.
  3. Migration loss: test real pre-0.29 fixtures and retain old preference documents.
  4. Platform leakage: capability checks—not global platform branches—own desktop/iOS differences.
  5. Native toolchain: repair the current React-Core-prebuilt CocoaPods metadata blocker before screenshot freeze.

Explicit review checkpoints

  1. Review the final RPC registry shape before runtime wiring.
  2. Approve fixture parity before migrating web selectors.
  3. Approve web/iPhone/iPad shells before thread presentation work.
  4. Review each workspace-tool slice independently.
  5. Review migration reports before deleting legacy.
  6. Approve validated DMG and iOS archive before any publication action.
