Stale active turns after Stop or restart

The custom fork inherited an upstream orchestration bug: a provider could become ready while its previous active turn ID remained attached, leaving the UI permanently “running” with no live work.

Upstream fix adopted 4,919 tests passed Typecheck passed Check passed

Provenance

This was not introduced by the quota UI, branding, desktop isolation, or Remote Mode changes.

  • The interrupt path is identical to upstream/main and originates in official T3 commits.
  • Official T3 fixed the defect in 501ce27b8: “Clear stale active turn when session becomes inactive” (#3159).
  • The custom release branched before that fix and therefore retained the older behavior.

Observed evidence

  • Three sessions reached ready while retaining non-null active_turn_id values.
  • Provider logs stopped mid-reasoning or mid-command and emitted no terminal turn event.
  • Restarting restored the contradictory persisted state instead of restoring the dead processes.
  • The SQLite database passed integrity checks; the defect was lifecycle reconciliation, not corruption.

Rollout and operational follow-up
